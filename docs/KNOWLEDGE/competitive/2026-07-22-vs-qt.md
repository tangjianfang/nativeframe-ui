## What Qt has
- Cross-platform: Win32, macOS, Linux, iOS, Android, WebAssembly.
- Signals/slots meta-object system (mature, moc-driven).
- QML declarative UI; rich ecosystem (3D, networking, SQL, multimedia).
- Commercial + LGPL dual license.

## What we have that Qt doesn't
- Pure Win32 — no abstraction layer over the OS GUI primitives.
- No moc build step; no qmake; just MSVC + CMake.
- Smaller binary footprint (a NativeFrameUI sample is ~150KB vs. Qt's several MB).
- Static linking for the entire stack (no DLL hell).

## What we explicitly choose NOT to clone
- Cross-platform abstraction (V1 non-goal; we're Win32-only).
- QML / declarative UI (V1 non-goal).
- Meta-object / signals-slots (V1 uses CommandRouter instead).

## Positioning
Qt is the "write once, run anywhere" choice. We're the "Win32 done right,
with modern C++ ergonomics and no abstraction tax" choice. Overlap is in
the IDE/sampling/sample-app surface; the production use case for each is
distinct.
