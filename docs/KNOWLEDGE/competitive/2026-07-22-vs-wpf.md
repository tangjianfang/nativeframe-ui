## What they have
- Managed (.NET) — automatic memory management, generics, LINQ bindings.
- XAML declarative markup.
- Hardware-accelerated rendering (Direct3D, formerly MILCore).
- Rich styling (templates, triggers, animations, transformations).
- Mature data binding (MVVM, INotifyPropertyChanged).
- Layout panels (StackPanel, Grid, DockPanel, Canvas).

## What we have that they don't
- Native C++20 (no CLR, no managed runtime, no .NET version coupling).
- Single-pass static linking — no runtime dependencies for distribution.
- Per-component STATIC library split.
- Zero GC pauses, deterministic destruction.
- Smaller binary footprint (~150KB vs. WPF's several MB plus CLR
  bootstrap).

## What we explicitly choose NOT to clone
- XAML declarative UI (V1 non-goal; we're imperative C++).
- Hardware-accelerated rendering via Direct3D (V2+ candidate — see
  P4.1).
- Full data binding (V1 non-goal; consumers wire callbacks manually).
- Layout panels beyond a basic hbox/vbox + splitter (V1.5+).

## Positioning
WPF is the "managed, declarative, hardware-accelerated Windows UI"
choice. We're the "native, imperative, lightweight Windows UI" choice.
Choose WPF if you're already in .NET. Choose NativeFrameUI if you're
in C++ and need deterministic perf + small footprint.
