## What MFC has
- Decades of accumulated control coverage: every common control + many niche ones.
- Deep integration with Visual Studio dialog/template editor.
- Document/view architecture primitives.
- Stable ABI; ships with every Windows Visual Studio install.

## What we have that MFC doesn't
- Modern C++20 throughout (MFC mixes C++ with legacy MFC macros/patterns).
- Zero dependency on Visual Studio tooling — pure CLI build via CMake presets.
- Smoke + boundary tests in CI that forbid MFC includes.
- Per-component library split.
- Free for any use.
- Smaller surface area (~15 controls vs. 250+).

## What we explicitly choose NOT to clone
- Doc/view (V1 non-goal).
- Template editor integration (V1 non-goal).
- ATL dependency (forbidden by boundary check).

## Positioning
MFC is the "Windows GUI standard for enterprise". We're the "modern Win32
for projects that can't or won't pull in MFC, want fine-grained component
control, or want theming without MFC's app-wizard-required infrastructure."
